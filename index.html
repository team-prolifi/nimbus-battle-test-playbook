Two different "prices" — don't confuse them: the purchase price (Set Pricing, step 3) is the money a customer pays to buy a credit unit; the redemption rate (Redemption Model, step 4) is a gate/display value used when spending credits — it is not a currency conversion.

Setup → Get Started (availability)

2

Setup → Entitlement Packages (purchasable quantities)

Packaging = Quantity → add three: 1,000 Credits · 5,000 Credits · 10,000 Credits (Display Name = the same text). Set 5,000 as default. These are the quantities a customer can buy; their money price comes from step 3.

3

Setup → Set Pricing — the price of a credit unit

This is the step we were missing. Choose a pricing model (credit schemes allow Per Unit · Volume · Tiered · Stair-Step) and set the money price per credit, per country. Canonical: Per Unit — 1 credit = 1 unit of local currency: NG ₦1 · US $1 · UK £1 · KE KSh 1 · DE €1 per credit (so a 5,000-credit pack costs 5,000 of that currency). To exercise a bulk discount instead, use Volume (e.g. 10,000+ credits at 5% off).

4

Setup → Redemption Model — one rate per currency

"What is the value of one unit of this prepaid credit…?" Add a Default Currency Redemption Rule for each currency, all 1 : 1, all enabled: 1 NGN = 1 Credit · 1 USD = 1 Credit · 1 GBP = 1 Credit · 1 KES = 1 Credit · 1 EUR = 1 Credit. A currency with no enabled rate can't be redeemed — that's the gate (display/gate only, not a conversion).

5

Setup → (optional) Customer Type Overrides

Add a customer-type-specific rate with an overdraft limit (e.g. Business customers, overdraft 5,000). This is the only place overdraft is enabled; without it a balance can never go negative.

6

Publish (Summary → Finish)

Draft → active. Only a draft scheme can be deleted. The scheme build ends here.

Accepting the credit is a product-side setting, not part of this build — a consuming product enables Accept pre-paid credit (+ compatible schemes) in its own general settings when you build that product. ⚠ Config-only — NOT enforced at redemption runtime (redemption ignores it): record as a finding, not a gate. Grant a balance, Redeem and Reverse are run as scenarios G1 · G2 · G3 (§05), not build steps.

Usage metering & event metrics — system-wide metering registry · build FIRST (metered features bind to a metric)

Prolifi has a general, merchant-facing metering system (Manage → Usage → Events / Metrics / Event Streams). It is not credit-specific — it measures any usage. The chain: an Event type (+ typed properties) defines what can happen → a Metric aggregates those events into a number (count · count_unique · sum · max · latest · custom, aggregating on a property) → Occurrences (event streams) are the "it happened" records → a metered feature is measured by pointing its Usage Metric at one. Build the event + metric for each metered feature before the feature (the feature's "Usage Metric" dropdown reads this registry).

Feature setup — detailed flow — availability · packages · pricing · run before mapping to any plan

After the bare create form, each feature has its own setup wizard (same Create → Setup pattern as products). Do all three before mapping it onto a plan — a feature can only be mapped where it's available and priced.

1

Setup → Availability (per country)

Turn on Global Availability + tick the countries the feature is sold in. Match the plans that will carry it — e.g. Cloud Storage & API Calls in all 5; Peak-Hours Access only where Business sells. Rule ①: a feature not available in a country cannot be mapped to a plan in that country (it won't appear / mapping is blocked there).

2

Setup → Entitlement Packages

Quantity features → the option tiles (Cloud Storage 50/100/250/500/Unlimited; default 100 GB). Range features → Minimum / Maximum (Support Hours 0–40; Bandwidth 0–5,000 GB). These are the shapes a plan's "Set Entitlements" then draws from.

3

Setup → Set Pricing (per country or global)

The scheme used to charge for extra capacity / enabling — separate from the plan fee. Allowed models: Boolean → Fixed only; Metered (allowance/consumption) → Per Unit · Volume · Tiered · Stair-Step. Enter a price per country, or a single global price. Build the named schemes from the list above (Storage — Per Unit, API Calls — Tiered, SMS — Stair-Step, SSO — Fixed Enable, Projects — Per Unit, Bandwidth — Tiered).

4

Then map per plan

With availability + packages + pricing set, the feature can be added to a plan and configured in Set Entitlements (Fixed / Variable / Pay-for-what-you-use) — see the overview + plan cards below.

🔎

Pricing-resolution rules to verify (at mapping / extension / checkout)

② Country price wins: where a per-country price exists, it's used and overrides the global — resolution is exact-country → global fallback. ③ No country price → global via FX: if a feature has no row for a country but has a global price, it resolves the local amount from the global price + exchange rate at mapping/checkout, not zero. These three (① not-available → not-mappable, ② country-wins, ③ FX-fallback) are the reflection checks for feature pricing — confirm each renders the right amount in the right currency.
